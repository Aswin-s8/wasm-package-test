01. Core Architecture

The Target-Oriented Compilation Model

Rust is an Ahead-Of-Time (AOT) compiled language. Instead of outputting bytecode for a VM (like Java or C#), the compiler (rustc) uses LLVM to generate highly optimized machine code specifically tailored to a target environment.

Target Triples: Rust identifies target environments using standardized triples in the format:
<arch>-<vendor>-<sys>-<abi>

Hover or click parts of the target triple on the right to dissect how Rust builds target specifications.

x86_64- pc- windows- msvc
Architecture (x86_64)
Specifies the CPU architecture, such as x86_64 (64-bit AMD/Intel), aarch64 (ARM64), or wasm32 (WebAssembly).
02. Native Builds

Native Targets: Powering the OS

Native compilation targets generate binaries that execute directly on the physical hardware under a specific host OS. This provides maximum speed, raw access to registers, and direct operating system features.

Windows (MSVC)
x86_64-pc-windows-msvc
Compiles 64-bit binaries for Microsoft Windows, linking against the Microsoft Visual C++ runtime and DLL dependencies.
Standard desktop & enterprise apps
Linux (GNU)
x86_64-unknown-linux-gnu
Compiles for Linux systems, dynamically linking the GNU C Library (glibc). Great for cloud containers and server environments.
Cloud servers & Docker images
macOS Darwin
aarch64-apple-darwin
Compiles native 64-bit binaries optimized for Apple Silicon (M1/M2/M3 chips) running macOS.
Apple hardware development
When to Choose Native: Use native compilation when interacting directly with host OS APIs (like filesystem access, direct threading, system audio/graphics APIs), executing high-performance server processes, or building standalone CLI applications.
03. Web Assembly

WebAssembly (Wasm): The Portable Target

WebAssembly (Wasm) is a low-level, assembly-like bytecode format designed to run with near-native speed in browser sandboxes. It represents a paradigm shift for web performance.

  • Target: wasm32-unknown-unknown
  • Safe Sandbox: Runs inside the web browser's isolated JS environment. It cannot access native system files or network drivers directly.
  • Integration: Interoperates seamlessly with JavaScript variables and DOM events via custom binding layers.
When to Choose Wasm: Use Wasm for complex web applications (e.g. video editors, Figma-like tools, 3D games), running Rust libraries on the frontend, or implementing edge serverless logic.
Target Wasm Blueprint
wasm32-unknown-unknown
wasm32
32-bit linear address space WebAssembly architecture.
unknown-unknown
No specific vendor runtime, and no target operating system API assumptions (a pure virtual environment).
Runs in V8, SpiderMonkey, Wasmtime
04. Build Setup

Configuring Rust Toolchains & Commands

Native targets require the host OS to supply a compatible linker (like MSVC Linker on Windows or GCC/Clang on Unix/macOS).

Terminal - Setup Native Target
# 1. Check currently installed toolchains and targets
$ rustup target list

# 2. Add the native target if compiling for a different OS architecture
$ rustup target add x86_64-pc-windows-msvc

# 3. Compile your project using Cargo pointing to target
$ cargo build --target x86_64-pc-windows-msvc --release

# 4. Find executable in: target/x86_64-pc-windows-msvc/release/app.exe

Wasm requires no system SDK/linker. To integrate with browsers conveniently, we use wasm-pack to automatically output JavaScript bindings.

Terminal - Setup WebAssembly
# 1. Download and register the WebAssembly compiler target triple
$ rustup target add wasm32-unknown-unknown

# 2. Compile directly to Wasm (produces raw target/wasm32-unknown-unknown/debug/lib.wasm)
$ cargo build --target wasm32-unknown-unknown

# 3. [Recommended] Build with wasm-pack to package automatically for web apps
$ wasm-pack build --target web

# 4. Result: A /pkg directory containing .wasm and auto-generated JS bindings (.js)

06. Summary

Native vs. WebAssembly Comparison

Feature Native Compilation WebAssembly Compilation
Target Triple x86_64-pc-windows-msvc, etc. wasm32-unknown-unknown
Sandbox & Security No Sandbox: Full OS/user authorization access. Sandboxed: Isolated memory space, safe in-browser execution.
OS & hardware access Direct (Filesystem, networking, device drivers). Indirect (Must bridge through JS/Web API imports).
Primary Outputs Executables (`.exe`, ELF binaries) or DLLs (`.dll`, `.so`). Bytecode binary modules (`.wasm`) & JS wrapper files.
